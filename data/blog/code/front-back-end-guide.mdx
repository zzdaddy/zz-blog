通常我们说，用node写后端服务有优势，是指的Node本身就是JS语法，省下了再去了解一门后端语言的学习时间。

但是这样真的对技术的深入有多大的好处吗？

我觉得最大的问题有两个：心态、思维方式。

# 够用就行？

在Vue、React兴起之初，大部分人学前端，是从基础语法、语言特性、JS原理学起的，基础还算扎实。

而现在框架发展到今天（2024年）这个地步，封装和魔改了太多细节（尤其是Vue），让一个新手上手去做，甚至不需要知道框架细节，只需要知道JS基本语法，然后知道框架的规则，写完后按照固定的命令去`build`，打包完后的`dist`扔在服务器上就完事。不清楚框架做了什么、解决了什么问题，也可以完成公司的任务。

在这种背景下，打上几年“螺丝”，逐渐能够应付公司所有的工作之后，对自己所使用的框架都是一知半解，怎么敢说对`JavaScript`有什么理解呢？

带着一知半解的JS基础开始写Node，**的确省去了学习语言的成本**。

但Node只是用JS语法实现了一套和系统交互的能力

最终用Node去开发，实现的是和其他后端语言一样的后端服务

你就会发现大量的问题，不知道后端服务如何组织代码，如何组织项目结构，有哪些要点需要提前架构。

**Node的api每个语法都能看懂，但是连在一起就是不知道如何正确使用。**

这时候你再去论坛不停的去搜：

_2024年了，用什么Node框架？_

_Node能否用在生产环境中？_

_哪些大公司在使用Node?_

**得到一些否定的答案之后，你再去各种技术群里去讨论：**

_大佬们，后端用什么啊，用Node有前途没有啊_

这个时候，你希望有个人出现，肯定一下Node的用途

然后这个人果真跳出来了，说我就是全套JS，开发多么多么快，赚了多少多少钱

于是，又像被打了鸡血一样，对Node又充满了希望

**别人是选了个实现产品最快的语言去赚钱，你是为了什么？**

就像纠结用TS还是JS一样

你觉得TS难用、不好学，于是四处去印证TS的说法，找到用JS的“同行”，然后心安理得的选择继续用JS

**我现在觉得这种给自己寻求自我安慰的心态大可不必**

找到安慰之后，你还是在摆弄自己的**小玩具**

而你所谓的“同道中人”，可能压根不关心语言的问题

可能人家关心的是 **语言的产物 -- 『产品』** 的转化、收入的问题？

所以

如果是为了技术的提升，为了真的去踏足后端的领域，就不要去纠结用不用Node或者转另一门语言

因为目的就是学习和深入，只要开始做了，在解决问题了，就会对自己有提升。

# 思维方式的转变

这里的思维方式，当然是指的是编码的思维

其实有点玄学了，所以我这里也只是表达自己的观点😏

**我觉得换一门语言，会对编码的思维方式产生极大的帮助**

因为JS的编码习惯已经养成了，再代入到后端开发中，短时间很难转换过来

比如

**很少考虑到抽象成类的问题，更多的讲究函数式编程，hooks等等**

**（小厂）很少考虑到日志的问题，因为不具备操作文件系统的能力，最多就是给日志服务上报**

**不考虑数据库、文件操作，所以对一些表结构、锁的问题、数据竞争等压根没有概念**

**多线程、任务等前端也不涉及，也是未知领域**

等等很多类似问题

在我继续用JS写后端服务时，会带有一些思维上的惯性：_我写JS的时候就是这么写的_

但是我一旦切换到其他强类型的后端语言，我发现就会好很多

因为我是抱着初学者的心态在学习，在吸收“前辈”的代码

因为语法到环境都是陌生的，所以我需要探索官方推荐怎么用，别人怎么用

因为我从思维上就在告诉自己这不是JS了，我需要用Go/Python/\*\*\*的思维去思考

而且不会去质疑为什么看不懂，因为这是新语言，不懂很正常，我要做的是用新的思维方式去理解为什么这么做

虽然一开始需要学习新语法，新语言特性，但是写着写着就会发现感受完全不同，**会让人有兴奋的感觉**

就像是电影《降临》里的讲述的一样，换一门语言也许真的能改变思维方式

# 结语

搞前端的想转后端，要看是想专注搞钱，还是深入学习

想搞钱 => 会什么用什么 => 什么快用什么

想学习 => 想用什么用什么 => 想法一定要落地
